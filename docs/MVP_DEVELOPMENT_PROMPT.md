# LogTrack Insight — MVP Development Prompt

> Copy everything below into your AI coding assistant (Claude Code, Cursor, etc.) as the project brief. It defines scope, stack, data model, folder structure, and security requirements for building the LogTrack Insight MVP.

---

## 1. Project Overview

Build LogTrack Insight, a QR-code-driven forest asset inventory and analytics system for DENR-PENRO Catanduanes. It replaces manual, paper-based tracking of confiscated logs, equipment (chainsaws/tools), and vehicles with a centralized, role-restricted, web-based platform.

The system tracks each confiscated asset from intake (MES) → Property custody → (if applicable) Property & Accounting/JEV → final disposition (donation, decay report, fabrication, release, forfeiture, or destruction), using QR codes to update status at each checkpoint.

This is an MVP: prioritize a working end-to-end vertical slice (auth → intake → QR tag → status lifecycle → dashboard → basic reports) over exhaustive edge-case handling. Flag anything descoped rather than silently skipping it.

---

## 2. Tech Stack (non-negotiable)

- Backend: PHP 8.2+, Laravel 11 (latest LTS-equivalent)
- Frontend: Laravel + Inertia.js + React (TypeScript) — gives an SPA-like experience while keeping Laravel as the single backend, and is required to use shadcn/ui properly (shadcn/ui is a React component system, not compatible with plain Blade)
- UI Library: shadcn/ui components on top of Tailwind CSS
- Database: MySQL 8+
- Auth: Laravel Breeze (Inertia + React starter kit) as the auth scaffold, extended with RBAC
- QR Codes: simplesoftwareio/simple-qrcode or endroid/qr-code (PHP) for generation; @zxing/library or html5-qrcode (JS) for browser-based scanning
- State/data fetching: Inertia's built-in page props + React Query only if needed for polling dashboard stats
- Charts (analytics dashboard): Recharts (pairs well with shadcn)
- PDF generation (acknowledgement receipts, JEV, ICS, PAR): barryvdh/laravel-dompdf or spatie/laravel-pdf

> Note: Chapter 1 mentions Laravel's templating engine for the UI. Since the requirement here is shadcn/ui + Tailwind, use Inertia + React instead of pure Blade — this is the standard, defensible way to pair Laravel with shadcn/ui and should be documented as a refinement in your methodology, not a contradiction.

## 3. Domain Flow (must be modeled exactly)

### 3.1 Intake — MES (Monitoring and Enforcement Section)

1. MES receives an asset under one of three modes: Apprehended, Abandoned, or Turned Over.
2. MES encodes: location apprehended, apprehending agency, asset type/species/equipment/vehicle details, turnover/abandonment info.
3. MES generates a printable Acknowledgement Receipt (system-generated form).

### 3.2 Property Custody

1. Property Custodian sees the encoded report on their dashboard.
2. Custodian verifies documentation completeness/correctness.
3. Custodian signs the Acknowledgement Receipt digitally (system) — must be signed before an item can be marked as stored.
4. Custodian generates/prints the asset's QR code.
5. Asset is tagged and placed in storage (physical placement is manual/out of scope — no GPS/location tracking).

### 3.3 Branch: Case Status

- If there's an ongoing case (Steady): asset stays in Property custody only — does not proceed to Accounting.
- If there's a Confiscation Order / Forfeiture Order: asset proceeds to Property and Accounting.

### 3.4 Property and Accounting

1. Accounting creates a JEV (Journal Entry Voucher) with a JEV number.
2. MES uploads/links the JEV to the asset record.
3. Asset status updates to "For Disposal", branching by asset type.

### 3.5 Documents generated by the system

- Acknowledgement Receipt (intake)
- JEV — Journal Entry Voucher
- ICS — Inventory Custodian Slip
- PAR — Property Acknowledgement Receipt
- Deed of Donation
- Decay Report
- Compliance/Inventory reports (DAO 97-32 quarterly reporting)

## 4. User Roles (RBAC — MVP scope)

Role
Key Permissions

System Admin
Full access, user/role management, audit logs

MES Officer
Encode intake, generate acknowledgement receipt, upload JEV, update case/disposal status

Property Custodian
Verify documentation, sign acknowledgement receipt, generate/print QR code, mark stored

Accounting Officer
Create JEV, generate ICS/PAR, process disposal documentation

PENRO Management / Regional Supervisor
Read-only dashboard, analytics, compliance report generation/export — no edit rights

Design roles as database-driven (roles/permissions tables), not hardcoded enums, so PENRO can adjust permissions without a code change later. Use Laravel Policies/Gates driven by a permissions pivot, or a package like spatie/laravel-permission (recommended for MVP speed).

## 5. Core Data Model (MVP)

Design migrations for at least:

- users (id, name, email, password, role_id, is_active, last_login_at, ...)
- roles, permissions, role_permissions (or use spatie/laravel-permission tables)
- assets (id, asset_code/uuid, type: enum[log, equipment, vehicle], species/description, municipality_of_origin, apprehending_agency, mode: enum[apprehended, abandoned, turned_over], has_ongoing_case: bool, current_status, qr_code_value, created_by, timestamps)
- asset_case_status_history (asset_id, status, changed_by, changed_at, notes) — full audit trail of status transitions
- acknowledgement_receipts (asset_id, signed_by_custodian_id, signed_at, pdf_path, receipt_number)
- jevs (asset_id, jev_number, created_by_accounting_id, uploaded_by_mes_id, pdf_path, created_at)
- disposals (asset_id, disposal_type: enum[donation, decayed, fabricated, released, forfeited, damaged], details JSON, report_pdf_path, processed_by, processed_at)
- donations (disposal_id, requester_name, deed_of_donation_path, released_at)
- ics_records, par_records (disposal_id, document_number, pdf_path, issued_by, issued_at)
- qr_scans (asset_id, scanned_by, scan_location_note, resulting_status, scanned_at) — audit trail of every scan, since scans drive status changes
- audit_logs (user_id, action, model_type, model_id, old_values JSON, new_values JSON, ip_address, created_at) — required for accountability/COA compliance angle described in Chapter 1
- documents (polymorphic: attachable_type, attachable_id, file_path, uploaded_by, uploaded_at) — supporting documents beyond the generated forms

Every status-changing action (sign receipt, scan QR, upload JEV, mark disposal) must write to asset_case_status_history and audit_logs. This is core to the system's value proposition (traceability/COA compliance) — do not treat it as optional.

## 6. MVP Feature Scope

In scope for MVP:

1. Authentication (login, logout, password reset) + RBAC-gated navigation and routes
2. Asset intake form (MES) with validation
3. Acknowledgement Receipt generation (PDF) + digital sign-off by custodian
4. QR code generation on custody confirmation; printable QR label
5. QR scan endpoint/page (camera-based, mobile-friendly) that updates status and logs the scan
6. Case-status branching logic (ongoing case vs. confiscation/forfeiture order)
7. JEV creation + upload/link to asset
8. Disposal workflows for logs (donation/decayed/fabricated), conveyance (release/forfeit), tools (damaged)
9. Real-time inventory dashboard: counts by status/type/municipality, recent activity feed
10. Basic analytics/reports: inventory summary, municipality-based confiscation stats, exportable compliance report (PDF/CSV)
11. Audit log viewer (admin only)
12. Responsive UI (desktop + mobile browser, since QR scanning happens on mobile devices)

Explicitly out of scope for MVP (per Chapter 1 limitations — do not build):

- GPS/physical location tracking within the storage facility
- Offline mode / offline QR scanning
- Field-level (ENRO) apprehension/verification workflows
- Hardware provisioning (QR printers, scanners) — assume browser camera + external label printer
- Legal/adjudication automation — the system only reflects status entered by staff, it doesn't decide cases

## 8. Security Requirements (mandatory for MVP, not "nice to have")

Given this stores law enforcement / government property records, treat these as hard requirements:

1. Authentication

- Laravel Breeze/Fortify hashed passwords (bcrypt/argon2, Laravel default)
- Enforce strong password policy (min length, complexity) via Password::defaults()
- Rate-limit login attempts (Laravel's built-in throttle middleware)
- Session-based auth (not stateless tokens) since this is a first-party web app — set SESSION_SECURE_COOKIE=true, SESSION_HTTP_ONLY=true in production
- Optional but recommended: 2FA for Admin/Management roles
2. Authorization (RBAC)

- Every controller action must check a Policy or Gate — never rely on hiding UI elements as the only access control
- Middleware-level route protection grouped by role
- Deny-by-default: new roles/permissions start with no access
3. Input validation & data integrity

- All input through FormRequest classes with explicit validation rules
- Use Eloquent parameter binding everywhere (no raw queries) to prevent SQL injection
- File uploads (supporting documents): validate MIME type + extension, store outside public/, serve via signed/authenticated route, virus-scan hook point if available
4. CSRF & session security

- Laravel's CSRF middleware stays enabled on all state-changing routes (Inertia handles this automatically — don't disable it)
- Regenerate session ID on login (session()->regenerate() — Breeze does this by default, verify it's intact)
5. XSS / output encoding

- React auto-escapes by default — never use dangerouslySetInnerHTML with user input
- Sanitize any rich text fields if introduced later
6. Audit logging

- Log every create/update/delete on assets, disposals, jevs, and every QR scan with user id, timestamp, IP, before/after values (see §5 audit_logs)
- Audit log itself should be append-only (no update/delete route exposed, even to Admin)
7. QR code integrity

- QR payload should be an opaque, unguessable token (e.g., signed UUID or Laravel signed URL), not a sequential asset ID — prevents enumeration/spoofing of assets by guessing QR values
- Scanning endpoint validates the token server-side before applying any status change
8. Transport & environment

- Enforce HTTPS in production (APP_URL https, FORCE_HTTPS via middleware or web server config)
- .env never committed; APP_DEBUG=false in production; generic error pages (no stack traces exposed)
- Configure security headers (CSP, X-Frame-Options, X-Content-Type-Options) — Laravel package spatie/laravel-csp or manual middleware
9. Database

- Dedicated MySQL user with least-privilege grants for the app (no root)
- Foreign key constraints enforced at DB level, not just app level, for referential integrity on audit-critical tables
10. Backups

- Since this is government property data, document a backup/restore plan for MySQL even in MVP (e.g., scheduled mysqldump), even if automation is deferred post-MVP

## 9. UI/UX Guidelines

- Use shadcn/ui primitives: Table (asset lists), Dialog/Sheet (forms, QR scan modal), Badge (status indicators — color-code by status), Card (dashboard stat tiles), Tabs (asset detail: Overview / Documents / History), Form (with react-hook-form + zod, shadcn's standard pairing)
- Status badges should be visually distinct and consistent across the app: e.g., under trial (blue), pending disposition (amber), cleared for donation/auction/disposal (green), forfeited (gray)
- Dashboard should surface, at minimum: total assets by type, assets by status, recent activity, municipality breakdown chart
- QR scan page must work well on a phone browser camera — large tap targets, clear success/error feedback after each scan
- All destructive/status-changing actions (sign receipt, confirm disposal) require a confirmation dialog

## 10. Suggested Build Order (MVP milestones)

1. Foundation: Laravel + Inertia + React + Tailwind + shadcn/ui scaffolding, MySQL connection, Breeze auth, roles/permissions seeder
2. Core data layer: migrations + models + policies for assets, JEVs, disposals, audit logs
3. Intake workflow: MES encode form → acknowledgement receipt PDF generation
4. Custody workflow: custodian dashboard, digital sign-off, QR code generation
5. Scan & lifecycle: QR scan page, status transition service, status history logging
6. Accounting branch: JEV creation/upload, case-status branching logic
7. Disposal workflows: logs (donation/decayed/fabricated), conveyance (release/forfeit), tools (damaged) + ICS/PAR generation
8. Dashboard & analytics: real-time counts, charts, filters
9. Reports: compliance/inventory report generation + export
10. Hardening pass: run through §8 security checklist explicitly before calling MVP "done"

## 11. Evaluation Alignment (for your ISO/IEC 25010 evaluation later)

Structure your build so these are demonstrably testable, since Chapter 1 commits to evaluating against them:

- Functional Suitability: each domain workflow in §3 maps to a testable feature
- Usability: keep forms short, use shadcn's accessible components, add inline validation messages
- Security: §8 checklist is your evidence base
- Reliability: status transitions should be atomic (DB transactions) so a failed step can't leave an asset in an inconsistent status
- Performance Efficiency: dashboard queries should be indexed (add DB indexes on assets.current_status, assets.type, assets.municipality_of_origin) and paginated, not loading full tables client-side
